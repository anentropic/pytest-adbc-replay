    **`cassette-format-rationale.md` — Cassette Format Rationale**

    Per diataxis Explanation rules: understanding-oriented, no step-by-step instructions, link to tutorials/how-tos for action items, discuss context and consequences.

    Cover three design questions:

    *Why files on disk rather than a database or network service?*
    - Cassettes need to live with the code — in the repo, committed to version control
    - A database or remote service would add operational overhead (another thing to run, back up, authenticate)
    - Files can be reviewed in pull requests, searched with grep, edited by hand in a pinch
    - The filesystem is the natural persistence layer for test data that travels with a codebase

    *Why Arrow IPC for query results?*
    - ADBC cursors return results as Arrow record batches — storing them in Arrow format is zero-cost serialisation (no conversion)
    - Arrow IPC preserves full schema metadata including column types, nullability, and dictionary encodings
    - JSON or CSV alternatives lose type information — a column of `Decimal` values becomes strings, `null` vs `0` ambiguity, date formatting edge cases
    - Arrow is language-agnostic: any language with an Arrow library can read a `.arrow` cassette file
    - The Arrow IPC file format (`RecordBatchFileWriter`) is stable and designed for on-disk storage (as opposed to the streaming format)

    *Why human-readable `.sql` files?*
    - The `.sql` file stores normalised SQL, not the raw query string as written in the test
    - Normalisation makes the key stable across whitespace and casing variations
    - Storing it as a plain `.sql` file means query changes appear as readable diffs in pull requests — a test that changes `SELECT id, name FROM users` to `SELECT id, name, email FROM users` shows exactly that change in the PR diff
    - This is the primary visibility feature: CI tests pass from cassettes, but humans can see what changed

    Close with a link to the [Cassette Format Reference](../reference/cassette-format.md) for the exact file naming convention.

    **`sql-normalisation-design.md` — SQL Normalisation Design**

    Cover three design questions:

    *Why cassette keys need normalisation*
    - The naive approach is to use the raw SQL string as the cassette key
    - This breaks as soon as a developer reformats the query: `SELECT * FROM foo` vs `select *  from foo` would be different keys, invalidating the cassette even though the query is semantically identical
    - Whitespace and case differences are a common source of spurious cassette misses in the absence of normalisation
    - Normalisation means the cassette key represents the query's intent, not its literal characters

    *What sqlglot does*
    - sqlglot parses the SQL into an AST and serialises it back to a canonical form
    - Canonical form: uppercase keywords, single spaces, consistent quote style
    - The output is also pretty-printed (`pretty=True`) which is what goes into the `.sql` file — so the human-readable file and the cassette key are the same text
    - When sqlglot cannot parse the SQL (vendor-specific syntax it does not understand), the plugin falls back to whitespace-only normalisation and emits a `NormalisationWarning`
    - The fallback still handles the most common variation (whitespace) without failing the test

    *Per-test dialect override*
    - Different databases use different SQL dialects — Snowflake, BigQuery, DuckDB each have syntax variants
    - sqlglot needs to know the dialect to parse correctly (e.g., Snowflake's `QUALIFY` clause, BigQuery's backtick quoting)
    - The global `adbc_dialect` ini key sets a project-wide default
    - The `dialect` argument on `@pytest.mark.adbc_cassette` overrides this per test — useful in multi-driver suites where different tests hit different databases
    - `None` or an empty string triggers sqlglot's auto-detect, which works for standard SQL

    Link to the [Configuration Reference](../reference/configuration.md) for the `adbc_dialect` setting.

    **`record-mode-semantics.md` — Record Mode Semantics**

    Cover the four modes and their intended use patterns:

    *Why four modes?*
    - A single `record: yes/no` flag would force re-recording everything each time you want new data
    - The four modes address different stages of a test suite's lifecycle

    *Mode-by-mode semantics*
    - `none`: the default. No warehouse connection is ever opened. Tests fail immediately with a clear error if the cassette is missing. Use this in CI and as the daily development default.
    - `once`: records a cassette the first time, then never again. Protects against accidental re-recording. Use when adding a new test to a suite where other cassettes already exist.
    - `new_episodes`: records only interactions not already in the cassette. Use when you've added new queries to an existing test — existing replay is untouched, new interactions are recorded.
    - `all`: re-records everything. Use when the live data has changed and you want to refresh all cassettes.

    *Common workflows*
    - **Initial recording**: first run uses `--adbc-record=once` to record all cassettes. Commit cassettes to version control. Switch back to `none` (the ini default) for all subsequent runs.
    - **Adding a query to an existing test**: use `--adbc-record=new_episodes` to record only the new interaction without disturbing existing cassettes.
    - **Refreshing stale cassettes**: use `--adbc-record=all` to re-record everything against the current live data.
    - **CI-only replay**: set `adbc_record_mode = none` in `pyproject.toml`. CI never records; local developers record when needed via CLI flag.
