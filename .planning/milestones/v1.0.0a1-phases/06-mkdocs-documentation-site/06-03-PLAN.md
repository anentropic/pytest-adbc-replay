    **Guide 4: `multiple-drivers.md` — Use multiple drivers in one session**
    Goal: wrap connections to different databases in the same test session.
    Content:
    - `adbc_replay` is session-scoped; `.wrap()` can be called multiple times on different connection objects
    - Show a conftest.py with two fixtures: one wrapping DuckDB, one wrapping a Snowflake connection (using `adbc_driver_snowflake`)
    - Explain that each wrapped connection records/replays independently — cassettes are keyed by SQL, not by connection
    - Driver-specific notes:
      - DuckDB: `adbc_driver_duckdb.dbapi.connect()` — no credentials, good for local dev/CI
      - Snowflake: `adbc_driver_snowflake.dbapi.connect(uri=...)` — credentials in env vars; use `once` mode locally, `none` in CI
      - BigQuery: similar pattern to Snowflake — connect via `adbc_driver_bigquery`
    - Note: SQL dialect normalisation is per-connection via the `adbc_dialect` ini key (global) or `@pytest.mark.adbc_cassette("name", dialect="snowflake")` per test

    **Guide 5: `scrub-sensitive-values.md` — Scrub sensitive values from cassettes**
    Goal: remove tokens, passwords, or PII from cassette files before they are written to disk.
    Content:
    - Explain when this matters: cassettes are committed to version control; if queries include credentials as parameter values, scrubbing prevents accidental exposure
    - Show how to implement a scrubber using the `adbc_scrubber` fixture:
      ```python
      @pytest.fixture(scope="session")
      def adbc_scrubber():
          def scrub(params):
              if params and "token" in params:
                  return {**params, "token": "REDACTED"}
              return params
          return scrub
      ```
    - Explain: the scrubber is called before parameters are written to the `.json` cassette file; return the modified params dict
    - Note: the scrubber applies to parameters only, not to the result Arrow table — Arrow results are stored as-is
    - Warning admonition: if the scrubbed cassette does not match the live query's parameters on replay, the cassette key lookup will fail — normalize scrubbed values consistently

    **Guide 6: `custom-param-serialisers.md` — Register custom parameter serialisers**
    Goal: handle parameter types that are not JSON-serialisable by default (e.g., numpy arrays, custom date wrappers).
    Content:
    - Explain: by default, parameters are serialised to JSON using the built-in encoder; types like `numpy.ndarray` or `datetime.date` subclasses will raise `TypeError`
    - Show how to register a custom serialiser via the `adbc_param_serialisers` fixture:
      ```python
      import numpy as np
      import pytest

      @pytest.fixture(scope="session")
      def adbc_param_serialisers():
          return {np.ndarray: lambda arr: arr.tolist()}
      ```
    - Explain: the fixture returns a `dict[type, Callable]`; the plugin calls the callable when it encounters that type during parameter serialisation
    - Note: the `NO_DEFAULT_SERIALISERS` sentinel (importable from `pytest_adbc_replay`) can be returned to disable all built-in serialisers and provide a fully custom set
    - Example using `NO_DEFAULT_SERIALISERS`:
      ```python
      from pytest_adbc_replay import NO_DEFAULT_SERIALISERS
      import pytest

      @pytest.fixture(scope="session")
      def adbc_param_serialisers():
          return NO_DEFAULT_SERIALISERS
      ```
